## 1. 記事の目的

- 「会議の議事録から、誰が・いつまでに・何をやるか」を自動抽出する処理を **Claude API + 構造化出力** で実装した知見を共有する。
- 単なるAPIコール紹介ではなく、**実プロダクト（MeetingMind AI）で使っているプロンプト本体・JSON Schema・失敗例と修正過程**まで全部出す。
- 読者が自分のプロダクトに「明日コピペで持っていける」レベルの再現性を担保する。

---

## 2. 各セクションの構成

### 2.1 はじめに（背景）
- HR出身の自分が、会議のアクションアイテム抜け漏れに何度も悩んだ実体験
- 既存の議事録ツールでは「タスク化」まで踏み込まない
- Claude を選んだ理由（長文コンテキスト・日本語の意図抽出・JSON出力安定性）

### 2.2 やりたいこと（インプット／アウトプット定義）
- インプット: 会議の文字起こしテキスト（〜1万字）
- アウトプット: `summary`, `decisions`, `actionItems[]`（`task`, `assignee`, `dueDate`, `priority`）
- なぜ最初に「型」を決めるのが最重要か

### 2.3 プロンプト設計の3原則（本記事の核）
1. **役割を狭く定義する**（"あなたは議事録アナリストです" ではなく "HR領域の議事録から行動責任を抽出する専任アナリスト" まで絞る）
2. **JSON Schema を system に固定し、user に会議テキストだけを渡す**
3. **few-shot は "失敗例 → 修正例" のペアで入れる**（成功例だけ並べても効果が薄い）

### 2.4 実装：システムプロンプト全文
- system プロンプト本体（日本語、約60行を全公開）
- 「担当者が "全員" や "未定" の場合の扱い」など、エッジケースをどう書き込んでいるか解説

### 2.5 実装：構造化出力（tool use / JSON mode）
- Claude の tool use を使って Schema を強制する方法
- `input_schema` の組み立てかた（Zod → JSON Schema 変換）
- パースに失敗したときのフォールバック戦略

### 2.6 実装：プロンプトキャッシュでコスト削減
- system プロンプトを `cache_control` で固定する
- 1会議あたりのコストが何円→何円に下がったか実測値
- キャッシュヒット率の確認方法

### 2.7 失敗集（読者が一番うれしい章）
- 「dueDate に "来週中" と返ってきてしまう」→ ISO8601 強制の書き方
- 「actionItems が空配列で返る」→ "なければ null ではなく根拠を書け" 指示
- 「敬語のせいで担当者名が "山田さん" になる」→ 後処理 vs プロンプト側で吸収する判断

### 2.8 評価：プロンプトを変えたら何が改善したか
- 評価データセット（社内会議10本）でのprecision/recall推移
- "改善した指標" だけでなく "悪化した指標" も載せる

### 2.9 まとめ・今後
- HRドメイン特化のプロンプトは "汎用議事録ツール" を超えられる、という確信
- 次回記事③（非エンジニアのロードマップ）への導線
